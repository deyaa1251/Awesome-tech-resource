## How to Contribute

1. **Fork the repository** to your GitHub account
2. **Create a new branch** for your contribution
   ```bash
   git checkout -b add-resource-name
   ```
3. **Make your changes** following the guidelines below
4. **Commit your changes** with a clear, descriptive message
   ```bash
   git commit -m "Add [Resource Name] to [Category]"
   ```
5. **Push to your fork**
   ```bash
   git push origin add-resource-name
   ```
6. **Submit a pull request** to the main repository

## Contribution Guidelines

### Resource Requirements

Your submission should meet the following criteria:

- **Relevance**: Directly related to tech learning, education, or professional development
- **Quality**: High-quality content that provides genuine educational value
- **Accessibility**: Preferably free or openly accessible (paid resources are acceptable if exceptional)
- **Active**: Recently updated or maintained (exceptions for timeless/archived resources of exceptional quality)
- **English**: Content should be primarily in English or have English translations
- **Legal**: No pirated content, copyright violations, or illegal material

### What Makes a Good Resource?

- **Educational Value**: Teaches concepts clearly and effectively
- **Depth**: Goes beyond surface-level explanations
- **Unique Perspective**: Offers something not easily found elsewhere
- **Community Validation**: Recognized or recommended by the tech community
- **Longevity**: Has proven valuable over time (or shows strong potential)

## Quality Standards

### Excellent Resources (Strongly Encouraged)

- In-depth technical blog posts with clear explanations
- Interactive tutorials with hands-on learning
- Comprehensive documentation and guides
- University-level course materials
- Research papers with practical applications
- Well-maintained open source projects
- Industry-recognized certifications and courses

### Acceptable Resources

- Curated lists and collections
- Podcasts with technical depth
- Video series with structured learning paths
- Technical newsletters with consistent quality
- Conference talks and presentations

### What We Don't Accept

- Spam or self-promotion without quality content
- Resources behind hard paywalls (without free tier)
- Outdated resources (unless historically significant)
- Duplicate entries
- Low-quality content farms
- Resources with intrusive ads or dark patterns
- Link shorteners or redirect URLs

## Formatting Guidelines

### Entry Format

Follow this format exactly:

```markdown
- [Resource Name](https://direct-url.com/) - Clear, concise description of what the resource offers.
```

### Description Guidelines

- Start with what the resource IS (blog, course, tutorial, etc.)
- Explain what topics it covers
- Mention the unique value it provides
- Keep it under 150 characters when possible
- Use active voice
- No marketing language or superlatives
- End with a period

### Examples

**Good:**
```markdown
- [Python Tutor](https://pythontutor.com/) - Visualize Python, Java, C++ code execution step-by-step.
- [Crafting Interpreters](https://craftinginterpreters.com/) - Complete guide to building interpreters from scratch.
```

**Bad:**
```markdown
- [Amazing Python Site](http://example.com) - The best Python learning resource ever!!!
- [Learn Code](http://example.com) - Website about coding
```

### Alphabetical Ordering

- Add new resources in alphabetical order within their category
- Alphabetize by the resource name, not the URL
- Ignore articles (a, an, the) when alphabetizing

## Pull Request Process

### Before Submitting

1. **Check for duplicates**: Search the list to ensure the resource isn't already included
2. **Test the link**: Verify the URL works and points to the correct resource
3. **Review formatting**: Ensure your entry matches the style guide
4. **One resource per PR**: Submit separate pull requests for multiple resources
5. **Choose the right category**: Place the resource in the most appropriate section

### PR Title Format

Use clear, descriptive titles:

```
Add [Resource Name] to [Category Name]
```

Examples:
- `Add Rust Book to Compilers & Programming Languages`
- `Add MIT OCW to Community Platforms & Interactive Learning`

### PR Description

Include the following in your pull request description:

```markdown
## Resource Information

**Name**: [Resource Name]
**URL**: [https://example.com]
**Category**: [Category Name]

## Why This Resource?

[Brief explanation of why this resource is valuable and meets the quality standards]

## Checklist

- [ ] Resource is relevant to tech learning
- [ ] Link is working and correct
- [ ] Entry follows the formatting guidelines
- [ ] Placed in alphabetical order within category
- [ ] No duplicate entry exists
- [ ] Description is clear and concise
```

### Review Process

- Maintainers will review your submission for quality and relevance
- You may be asked to make changes or provide additional context
- Be patient and responsive to feedback
- PRs are typically reviewed within 1-7 days

## What to Avoid

### Self-Promotion

- If you're submitting your own content, disclose this in the PR
- Your content should meet the same high standards as other submissions
- Focus on the value provided, not self-promotion

### Commercial Content

- Paid resources are acceptable if they offer exceptional value
- Clearly note if something requires payment
- No affiliate links or referral codes
- No resources that are purely commercial without educational content

### Maintenance

- Don't submit resources you don't personally use or haven't reviewed
- If you notice a broken link or outdated resource, submit a PR to remove or update it
- Help keep the list current and valuable
